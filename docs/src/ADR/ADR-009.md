# ADR-009 - Utilisation d'un monorepo

- **Statut** : Accepté
- **Date** : 2026-04-03

## Contexte

Le projet comporte plusieurs applications (`web`, `server`) et packages partagés (`api`, `auth`, `db`).
Ces composants sont fortement couplés et évoluent ensemble : un changement dans le schéma de base de données impacte simultanément le backend et le frontend.
Il est donc nécessaire de choisir une stratégie d'organisation du code source.

## Décision

Nous choisissons d'organiser le projet sous forme de monorepo géré avec Turborepo et Bun workspaces.

## Alternatives considérées

- **Polyrepo** : un dépôt Git par application/package — versioning indépendant mais synchronisation complexe entre dépôts
- **Monorepo sans outil de build** : organisation simple mais pas de cache ni de pipeline de build optimisé

## Conséquences

- Partage de code facilité entre les packages (`api`, `auth`, `db`) et les applications
- Refactoring atomique : un seul commit peut modifier frontend, backend et packages partagés
- Cache de build et exécution parallèle des tâches via Turborepo
- Configuration unifiée (TypeScript, Biome, tests) à la racine du dépôt
- Dépôt potentiellement plus lourd à cloner pour un contributeur ne travaillant que sur une partie
